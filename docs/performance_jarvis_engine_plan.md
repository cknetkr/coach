# 수행평가 자비스 엔진 기획안

## 1. 프로젝트 정의

### 1.1 프로젝트명
수행평가 자비스 엔진  
Performance Jarvis Engine

### 1.2 한 줄 정의
학생이 수행평가 문서, 루브릭, 안내문, 예시 자료를 넣으면  
AI가 이를 해석해 과제 구조를 이해하고, 절차를 스스로 만들고, 단계별로 코칭하며, 최종 제출물까지 완성하도록 돕는 자율형 수행평가 엔진

### 1.3 핵심 목표
기존처럼 사람이 수행평가마다
- 단계 구조를 미리 짜고
- 입력 폼을 직접 설계하고
- 프롬프트를 유형별로 따로 만드는 방식에서 벗어나

AI가 문서를 읽고 스스로
- 과제 형식을 파악하고
- 절차를 생성하고
- 질문 순서를 정하고
- 결과물을 검증하는 시스템을 만든다.

## 2. 왜 이 엔진이 필요한가

현재 수행평가 도구들은 대부분 두 가지 한계를 가진다.

1. 생성 중심이다
- 글을 써 주기는 하지만
- 무엇부터 해야 하는지
- 어떤 순서로 진행해야 하는지
- 어떤 조건을 놓치면 감점되는지
를 체계적으로 관리하지 못한다.

2. 고정형이다
- 수행1, 수행2, 수행3처럼 미리 정해진 유형만 처리하기 쉽다.
- 문서 형식이 바뀌면 로직도 다시 수정해야 한다.

하지만 실제 학교 수행평가는 과목마다, 학교마다, 심지어 같은 과목 안에서도 형식이 다르다.

예:
- 독서주제탐구
- 비평문
- 발표 대본
- 실험 보고서
- 토론 개요서
- 포트폴리오
- 프로젝트 산출물

따라서 필요한 것은
**정답 생성기**가 아니라  
**문서를 읽고 절차를 만들어 주는 엔진**이다.

## 3. 제품 철학

### 3.1 생성보다 진행
이 엔진의 본질은 결과물을 한 번에 만들어 주는 것이 아니라,
과제를 수행 가능한 단계로 바꾸고,
학생이 그 단계를 따라가며 완성할 수 있게 돕는 것이다.

### 3.2 고정형보다 해석형
이 엔진은 특정 수행평가 탭을 미리 만들어 두는 구조가 아니라,
입력된 문서에서 요구사항을 추출해
그 과제에 맞는 구조를 새로 만든다.

### 3.3 대필보다 코칭
이 엔진은 대신 써 주는 시스템이 아니라,
학생이 생각을 정리하고,
근거를 보완하고,
루브릭을 충족하는 방향으로 완성하도록 돕는 코치여야 한다.

### 3.4 과목성 + 문서성을 함께 본다
수행평가는 과목마다 좋은 절차가 다르다.
하지만 같은 과목 안에서도 과제 형식은 다르다.

따라서 최적 구조는
- 과목 전문성
- 문서 해석 능력

을 함께 쓰는 하이브리드 구조다.

## 4. 기존 SelfQ 분석에서 얻은 인사이트

현재 `selfq`의 수행평가 구조는 이미 중요한 기반을 갖고 있다.

### 4.1 이미 잘 되어 있는 점
- 수행평가를 일반 시험과 분리해 전용 플로우로 다룸
- 과목별 페르소나를 둬서 국어, 영어, 수학, 과학 등의 진행 방식 차이를 반영함
- Gemini가 `steps`, `questions`, `constraints`를 생성하는 구조가 있음
- 단계별 질문과 단계별 결과물 빌드업 흐름이 존재함
- 최종 초안, 체크리스트, 루브릭, 코칭 분리 구조가 있음

### 4.2 최근 국어 업데이트가 보여 준 것
최근 국어 업데이트는 다음 방향이 중요하다는 것을 보여 준다.

- 국어 수행평가는 단순 감상문이 아니라 절차형 프로젝트다.
- 좋은 흐름은 `도서 분석 → 사회 문제 매핑 → 개요 작성 → 초안 작성 → 조건 검증`이다.
- 과목별 페르소나는 실제로 절차 품질에 큰 영향을 준다.
- 단계 결과를 `fullDraft`, `stepResults`, `finalComment`까지 저장하는 방식은 매우 유용하다.

### 4.3 현재 한계
하지만 현재 구조는 여전히
`과목별 기본 프로세스`에 많이 의존한다.

즉,
- 국어면 비평문 중심
- 영어면 PREP 중심
- 수학이면 풀이 절차 중심

으로 흐르기 쉽다.

이 방식은 강력하지만,
문서가 요구하는 형식이 과목 템플릿과 다를 때
절차를 완전히 새로 발명하기는 어렵다.

## 5. 엔진의 핵심 문제 정의

이 엔진이 해결해야 할 본질적인 문제는 다음이다.

### 문제 1. 문서를 사람이 해석하고 있다
현재는 사용자가
- 이 수행평가가 어떤 형식인지
- 무엇을 먼저 해야 하는지
- 어떤 항목이 필수인지
를 어느 정도 이해한 뒤 AI를 써야 한다.

### 문제 2. 절차가 하드코딩되어 있다
과제 형식이 바뀌면
- UI 수정
- 프롬프트 수정
- 로직 수정
이 필요하다.

### 문제 3. 과목은 반영하지만 문서 형식은 덜 반영한다
국어 과제라고 해도
- 독서기록
- 발표문
- 비평문
- 감상문
은 절차가 달라야 한다.

## 6. 엔진 비전

이 엔진은 아래 흐름을 자동화해야 한다.

`문서 입력 → 문서 해석 → 요구사항 구조화 → 절차 생성 → 단계별 코칭 → 결과 검증 → 제출용/훈련용 출력`

즉 사용자는
“이 수행평가 해야 해”
라고 문서를 넣기만 하면 되고,
엔진은 다음을 수행해야 한다.

1. 과제 구조 파악
2. 형식 분류
3. 필수 항목 추출
4. 제약 조건 추출
5. 단계 순서 생성
6. 학생에게 필요한 질문 생성
7. 각 단계 결과물 생성
8. 루브릭 검증
9. 최종 제출본 및 훈련 자료 생성

## 7. 엔진 아키텍처

## 7.1 전체 구조

### Layer 1. Input Layer
입력:
- 수행평가 안내문
- 루브릭
- 예시 답안
- 교사 피드백
- 첨부 PDF
- 이미지
- 텍스트

역할:
- 문서 수집
- OCR 또는 텍스트 추출
- 파일 유형 정리

### Layer 2. Document Parser
역할:
- 문서를 읽고 요구사항 구조를 추출

추출 대상:
- 과제명
- 과목명
- 수행평가 유형
- 필수 항목
- 세부 항목 수
- 순서
- 글자 수 조건
- 제출 형식
- 평가 기준
- 금지 조건
- 참고자료 요구
- 발표/시험 여부

예:
- 인상적인 문구 2개
- page 번호 필수
- 내용 겹치면 안 됨
- 1500±50자
- 5문단
- 자료 반입 불가

### Layer 3. Assignment Schema Builder
역할:
- 파싱된 내용을 공통 구조 JSON으로 변환

예시 구조:

```json
{
  "assignmentName": "독서주제탐구",
  "subjectName": "국어",
  "assignmentType": "reading-analysis",
  "constraints": [
    "인용문 2개",
    "page 번호 필수",
    "서로 다른 내용",
    "책 전체 흐름이 드러나야 함"
  ],
  "steps": [
    {
      "id": "reason",
      "title": "작품 선정 이유",
      "required": true
    }
  ]
}
```

이 구조는 사람이 쓰는 것이 아니라
엔진이 자동 생성해야 한다.

### Layer 4. Persona Layer
역할:
- 과목 특성에 맞는 사고 흐름과 코칭 스타일을 보강

예:
- 국어: 작품 분석, 사회 문제 연결, 문단 구조 강조
- 영어: PREP, 원어민 표현, 말하기 훈련 강조
- 수학: 풀이 과정, 중간 계산, 검산 강조
- 과학: 가설, 변인 통제, 결과 해석 강조

이 레이어는 절차를 결정하는 주 엔진이 아니라,
절차를 더 똑똑하게 만드는 보조 엔진이다.

### Layer 5. Workflow Generator
역할:
- Assignment Schema + Persona를 조합해 실제 단계 구조 생성

출력:
- phase 목록
- step 목록
- 각 step의 질문
- 통과 조건
- 단계별 산출물 형식

### Layer 6. Guided Tutor
역할:
- 실제 학생과 상호작용
- 한 단계씩 질문
- 입력이 부족하면 추가 질문
- 다음 단계 안내
- 감점 위험 경고

### Layer 7. Build Engine
역할:
- 각 단계 결과물을 생성
- 이전 단계 결과를 이어받아 누적 빌드
- 통합 초안 생성

### Layer 8. Rubric Checker
역할:
- 결과물이 과제 문서 조건을 만족하는지 자동 점검

검사 예:
- 개수 충족
- 글자 수 범위
- 문단 수
- 출처 표기
- 인용문 중복
- 필수 항목 누락

### Layer 9. Output Composer
역할:
- 최종 결과물을 목적별로 정리

출력 종류:
- 제출용 결과물
- 발표용 대본
- 체크리스트
- 암기 카드
- 재현 훈련본

## 8.3 예시: 독서주제탐구

문서 해석 결과:

- 줄거리
- 태도 변화
- 사회 현실 연결

생성 절차:

4. 생각 변화 정리
5. 경험·사회 현실 연결
6. 인용문 2개 수집 및 분석

## 9. 기존 방식과 새 엔진의 차이

### 9.1 기존 방식
- 과목별 절차를 먼저 정의
- 문서는 그 절차를 구체화하는 자료
- 수행평가 형식이 바뀌면 수정 비용이 큼

### 9.2 새 엔진 방식
- 문서를 먼저 해석
- 문서에서 과제 구조를 추출
- 그 결과로 절차를 생성
- 과목 페르소나는 보강 레이어로 사용

즉,
기존 방식은 `과목 중심`
새 엔진은 `문서 중심`
이다.

## 10. 왜 과목 페르소나를 버리면 안 되는가

문서 중심 엔진만으로는 부족하다.

이유:
- 같은 문서 형식이어도 과목별 좋은 답변 방식이 다르다.
- 국어의 “좋은 비평문 구조”와 수학의 “좋은 서술형 풀이 구조”는 완전히 다르다.
- 질문 방식과 피드백 방식도 과목별로 달라야 한다.

따라서 정답 구조는

`문서 중심 구조 생성 + 과목 중심 전문성 보강`

이어야 한다.

## 11. 핵심 데이터 모델

```json
{
  "input": {
    "subjectName": "국어",
    "documents": ["rubric.pdf", "guide.txt"]
  },
  "schema": {
    "assignmentType": "critical-essay",
    "constraints": ["1500±50자", "5문단", "사회 문제 연결"],
    "sections": ["도입", "전개1", "전개2", "전개3", "정리"]
  },
  "persona": {
    "title": "수석 논술 코치",
    "processStyle": ["작품 분석", "사회 문제 연결", "초안", "검증"]
  },
  "workflow": {
    "steps": [
      {
        "id": "book_analysis",
        "title": "작품 분석",
        "questions": []
      }
    ]
  },
  "result": {
    "draftSections": [],
    "fullDraft": "",
    "checklist": [],
    "finalComment": ""
  }
}
```

## 12. 제품 기능 설계

### 12.1 문서 분석 기능
- PDF, 이미지, 텍스트 입력
- 조건·항목·루브릭 추출
- 문서 요약 및 감점 포인트 파악

### 12.2 절차 자동 생성 기능
- 단계 수 자동 결정
- 단계별 질문 생성
- 최소/필수 단계 구분

### 12.3 단계별 코칭 기능
- 학생의 현재 답변 수준 판별
- 부족하면 보완 질문
- 다음 단계 이동 전 체크

### 12.4 초안/최종본 생성 기능
- 단계 결과 누적
- 통합 초안 조립
- 최종본 포맷팅

### 12.5 검증 기능
- 글자 수
- 문단 수
- 필수 항목
- 중복
- 출처
- 감점 위험

### 12.6 훈련 기능
- 발표 대본 분해
- 암기 카드
- 빈칸 문제
- 재현 테스트

## 13. 사용자 경험 설계

### 13.1 온보딩
- 과제 문서 업로드
- 과목 자동 추정
- 과제 형식 자동 추출
- “AI가 이해한 과제 구조”를 먼저 보여줌

### 13.2 진행 화면
- 좌측: 현재 단계

### 13.3 완료 화면
- 제출용 결과물
- 체크리스트
- 보완 포인트
- 훈련 시작 버튼

목표:

- step/questions/constraints 구조 표준화

## 17. 최종 결론

수행평가 자비스 엔진의 핵심은
예쁜 입력창이 아니다.

핵심은 다음 다섯 가지다.

1. 문서를 읽는 능력
2. 요구사항을 구조화하는 능력
3. 절차를 생성하는 능력
4. 단계별로 코칭하는 능력
5. 마지막에 조건을 검증하는 능력

현재 `selfq`는 이미 과목 중심 수행평가 코치로서 강한 기반을 갖고 있다.
특히 최근 국어 업데이트는 좋은 사례다.

다음 단계는 그 기반을 버리는 것이 아니라,
그 위에 `문서 중심 절차 생성 엔진`을 올리는 것이다.

한 줄로 정리하면:

**수행평가 자비스 엔진은  
과목별 전문성을 가진 AI가  
문서를 읽고 스스로 절차를 설계해  
학생을 끝까지 이끄는 자율형 학습 엔진이다.**

## 18. 과목별 엔진 보강 방향

현재 국어는 최근 업데이트를 통해
- 절차
- 페르소나
- 질문 방식
- 초안 흐름
- 검증 포인트

가 비교적 풍성하게 설계되었다.

반면 수학과 영어는 아직 더 구체화할 여지가 있다.
이 엔진이 진짜 자비스가 되려면,
수학과 영어도 국어처럼
**“과목 특유의 사고 흐름이 보이는 수준”**
으로 보강되어야 한다.

## 19. 수학 수행평가 자비스 엔진 보강안

### 19.1 수학 수행평가의 본질
수학 수행평가는 단순히 정답을 맞히는 과제가 아니라,

- 풀이 과정의 논리성
- 조건 해석
- 수식 전개
- 서술형 설명
- 오류 검산

을 평가하는 과제다.

특히 수행평가에서는 객관식보다
`왜 그렇게 풀었는지`를 드러내는 과정이 중요하다.

### 19.2 수학 엔진이 반드시 가져야 할 흐름

국어가
`작품 분석 → 사회 문제 연결 → 초안 → 검증`
이라면,

수학은 다음 구조가 더 적합하다.

1. 문제 구조 해석
- 주어진 조건 정리
- 구해야 하는 값 명시
- 숨겨진 조건 또는 함정 파악

2. 핵심 개념 매핑
- 어떤 공식/정리/단원을 써야 하는지 식별
- 왜 그 개념이 필요한지 설명

3. 풀이 전략 설계
- 바로 계산하지 않고
- 어떤 순서로 풀 것인지 결정

4. 연산 및 전개
- 식 전개
- 계산 과정
- 중간 과정 표시

5. 검산 및 표현 보정
- 계산 오류 확인
- 단위·기호 확인
- 서술형 문장 정리

### 19.3 수학 전용 절차 예시

#### 유형 A. 서술형 풀이 수행평가
1. 문제 조건 해석
2. 관련 개념/공식 선택
3. 풀이 순서 설계
4. 풀이 전개
5. 최종 답 및 검산

#### 유형 B. 탐구형 수학 수행평가
1. 주제 정의
2. 수학 개념 연결
3. 사례 또는 데이터 수집
4. 규칙성 분석
5. 결론 및 해석

#### 유형 C. 실생활 연계형 수학 보고서
1. 실제 문제 상황 정리
2. 수학 모델 설정
3. 식·그래프·표 구성
4. 계산 결과 해석
5. 현실 적용 의미 정리

### 19.4 수학 페르소나 보강 방향

현재 수학 페르소나는 풀이 과정과 감점 함정을 중시하는 방향으로 좋다.
여기에 아래가 더 필요하다.

- `문제 유형 분류 능력`
  - 계산형
  - 설명형
  - 증명형
  - 탐구형
  - 실생활 적용형

- `서술형 표현 코칭`
  - 단순 계산이 아니라 설명 문장 교정
  - “왜 이 공식을 썼는지”를 쓰게 유도

- `검산 루프`
  - 최종 답 이전에 검산 단계 강제

- `오개념 탐지`
  - 학생이 자주 빠지는 논리 비약 자동 감지

### 19.5 수학 결과물 모델

수학은 국어처럼 `fullDraft`만 있으면 부족하다.
추가로 아래 구조가 필요하다.

- `conditionSummary`
  - 조건 요약
- `conceptMap`
  - 사용 개념/공식
- `solutionSteps`
  - 단계별 풀이
- `finalAnswer`
  - 최종 정답
- `verification`
  - 검산 내용
- `teacherStyleExplanation`
  - 제출용 서술 문장

### 19.6 수학 검증 엔진 강화 포인트

- 계산 과정 누락 여부
- 공식 적용 타당성
- 기호 사용 오류
- 단위 누락
- 최종 답 형식 오류
- 논리 비약 여부

## 20. 영어 수행평가 자비스 엔진 보강안

### 20.1 영어 수행평가의 본질
영어 수행평가는 단순 번역이나 문장 생성이 아니라,

- 의사 전달력
- 문법 정확성
- 표현 자연스러움
- 발화 구조
- 주제 일관성

을 종합적으로 평가하는 과제다.

특히 영어 수행평가는 유형 차이가 크다.

- 영어 에세이
- 발표 대본
- 자기소개
- 인터뷰 대비
- 받아쓰기
- 영어 독후감
- 토론 스크립트

따라서 국어처럼 한 흐름으로 묶기보다
형식별 세부 분기가 더 중요하다.

### 20.2 영어 엔진이 반드시 가져야 할 흐름

영어 수행평가 기본 구조는 다음이 적합하다.

1. 과제 형식 파악
- 에세이인지
- 발표인지
- 면접인지
- 받아쓰기인지
- 토론인지

2. 핵심 메시지 설계
- 전달해야 할 주장/내용 정리
- 한글 사고 정리

3. 영어 구조 설계
- PREP
- introduction-body-conclusion
- question-answer
- opinion-support

4. 표현 생성 및 교정
- 문법
- 어휘
- 자연스러운 표현
- 연결어

5. 말하기/듣기 대응 보강
- 발음 포인트
- 강세
- 꼬리 질문 대응
- 받아쓰기 위험 표현 정리

### 20.3 영어 전용 절차 예시

#### 유형 A. 영어 에세이 수행평가
1. 주제 해석
2. 핵심 주장 정리
3. PREP 또는 5문단 구조 설계
4. 초안 작성
5. 문법/표현 교정

#### 유형 B. 영어 발표 수행평가
1. 발표 목적 정리
2. 발표 흐름 구성
3. 말하기용 문장 변환
4. 발음·강세 포인트 정리
5. 예상 질문 대비

#### 유형 C. 영어 면접/구술
1. 질문 유형 분류
2. 답변 포인트 정리
3. 핵심 표현 생성
4. 짧고 말하기 쉬운 문장으로 정리
5. 꼬리 질문 방어

#### 유형 D. 받아쓰기/청취형
1. 핵심 문장 추출
2. 연음·축약형 분석
3. 빈출 표현 정리
4. 반복 훈련 포인트 생성

### 20.4 영어 페르소나 보강 방향

현재 영어 페르소나는 PREP 구조와 원어민 교정을 강조하는 방향으로 좋다.
여기에 아래를 더 강화해야 한다.

- `유형 분기 강화`
  - 영작
  - 발표
  - 면접
  - 받아쓰기
  - 토론

- `한국어 사고 → 영어 구조 변환`
  - 학생이 먼저 한국어로 논리를 잡고
  - 그걸 영어 구조로 바꾸게 해야 함

- `말하기 친화적 출력`
  - 읽기용 문장과 말하기용 문장을 구분

- `발화 후속 훈련`
  - 강세
  - 끊어 읽기
  - 꼬리 질문
  - 예상 오류

### 20.5 영어 결과물 모델

영어는 국어처럼 초안만 있으면 부족하고,
수학처럼 풀이 단계도 아니다.
다음 구조가 적합하다.

- `messageMap`
  - 전달하고 싶은 핵심 메시지
- `structurePlan`
  - PREP 또는 발표 구조
- `englishDraft`
  - 영어 초안
- `naturalRevision`
  - 자연스러운 표현 교정본
- `speakingVersion`
  - 말하기용 짧은 문장 버전
- `pronunciationGuide`
  - 강세/연음 포인트
- `followUpQA`
  - 예상 질문과 답변

### 20.6 영어 검증 엔진 강화 포인트

- 문법 오류
- 어휘 수준
- 표현 자연스러움
- 주제 일관성
- 문단 연결성
- 발표 시간 적합성
- 발화 난이도

## 21. 국어·수학·영어 3과목 공통 비교

### 21.1 국어
- 핵심: 해석과 논증
- 좋은 절차: 작품 분석 → 사회 문제 연결 → 초안 → 조건 검증
- 결과물 중심: 비평문, 감상문, 발표문

### 21.2 수학
- 핵심: 조건 해석과 논리 전개
- 좋은 절차: 조건 정리 → 개념 선택 → 풀이 설계 → 계산 전개 → 검산
- 과정 중심: 서술형 풀이, 탐구형 분석

### 21.3 영어
- 핵심: 메시지 전달과 언어 표현
- 좋은 절차: 과제 형식 파악 → 논리 설계 → 영어 구조화 → 표현 교정 → 발화 훈련
- 표현 중심: 영작, 발표, 면접, 듣기

## 22. 엔진 설계에 반영해야 할 공통 원칙

수학과 영어를 국어처럼 풍성하게 만들기 위해서는
단순히 example topic 몇 개를 넣는 것으로는 부족하다.

아래 4가지를 과목마다 갖춰야 한다.

1. `과목별 processSteps`
- 과목 특유의 사고 순서가 명확해야 한다.

2. `과목별 결과물 모델`
- 국어는 fullDraft,
- 수학은 solutionSteps,
- 영어는 speakingVersion 같은
과목별 핵심 산출물이 달라야 한다.

3. `과목별 검증 엔진`
- 국어는 문단/논지,
- 수학은 풀이과정/검산,
- 영어는 문법/자연스러움/발화성

4. `과목별 후속 훈련`
- 국어: 발표/암기
- 수학: 유사문제 재현
- 영어: 발화/면접/받아쓰기

## 23. 보완 후 최종 방향

이 엔진은 국어만 강한 시스템이 되어서는 안 된다.

최종적으로는

- 국어는 논술형 사고 엔진
- 수학은 풀이형 사고 엔진
- 영어는 표현형 사고 엔진

처럼 작동해야 한다.

즉, 같은 “수행평가 자비스” 안에서도
과목마다 자비스의 사고 방식이 달라져야 한다.

한 줄로 정리하면:

**국어처럼 풍성하다는 것은  
예시가 많다는 뜻이 아니라,  
그 과목이 생각하는 방식 자체가 엔진에 들어가 있다는 뜻이다.**

## 25. 수학 processSteps 실제 문구 초안

아래 문구는 `starterTemplate`, `actionNote`, `step.goal`, `step.action` 생성에 바로 활용할 수 있는 형태다.

### 25.1 Step 1. 문제 조건 및 요구사항 해석
- 목표: 문제에서 주어진 조건과 구해야 할 값을 명확히 분리한다.
- 액션: 식별 가능한 조건, 숨은 조건, 단위, 범위를 먼저 정리한다.
- 튜터 질문 예시:
  - 이 문제에서 반드시 써야 하는 조건은 무엇인가요?
  - 최종적으로 구해야 하는 값은 하나인가요, 여러 개인가요?
  - 조건 중 빠뜨리기 쉬운 부분이 있나요?

### 25.2 Step 2. 핵심 개념·공식 선택
- 목표: 어떤 단원 개념과 공식을 써야 하는지 확정한다.
- 액션: 공식 이름만 적지 말고 왜 이 공식을 선택했는지 설명한다.
- 튜터 질문 예시:
  - 이 문제는 어떤 단원의 개념으로 보는 것이 가장 적절한가요?
  - 다른 공식이 아니라 이 공식을 선택한 이유는 무엇인가요?

### 25.3 Step 3. 풀이 전략 설계
- 목표: 계산 전에 풀이 순서를 먼저 설계한다.
- 액션: 바로 계산하지 말고 1단계, 2단계, 3단계 순서를 세운다.
- 튜터 질문 예시:
  - 가장 먼저 구해야 하는 중간값은 무엇인가요?
  - 식을 어떻게 세우면 계산이 단순해지나요?

### 25.4 Step 4. 연산 및 단계별 전개
- 목표: 중간 과정을 생략하지 않고 풀이를 전개한다.
- 액션: 식 전개, 치환, 계산 근거를 단계별로 적는다.
- 튜터 질문 예시:
  - 여기서 식이 왜 이렇게 바뀌는지 한 줄로 설명할 수 있나요?
  - 중간 과정을 생략하면 감점될 부분은 어디인가요?

### 25.5 Step 5. 검산 및 서술형 표현 보정
- 목표: 계산과 논리를 다시 점검하고 채점 가능한 답안으로 정리한다.
- 액션: 최종 답 검산, 단위 확인, 서술 문장 정리를 수행한다.
- 튜터 질문 예시:
  - 최종 답이 조건과 모순되지 않나요?
  - 채점자가 보기 쉽게 마지막 문장을 어떻게 정리할 수 있나요?

## 26. 영어 processSteps 실제 문구 초안

### 26.1 Step 1. 과제 형식 및 전달 목적 파악
- 목표: 이 과제가 에세이인지, 발표인지, 면접인지, 받아쓰기인지 확정한다.
- 액션: 평가 형식과 전달해야 할 핵심 목적을 먼저 분리한다.
- 튜터 질문 예시:
  - 이 과제는 읽히는 글인가요, 말하는 대본인가요?
  - 핵심 메시지를 한 문장으로 말하면 무엇인가요?

### 26.2 Step 2. 핵심 메시지와 논리 구조 설계
- 목표: 영어로 쓰기 전에 전달 논리를 먼저 세운다.
- 액션: 한국어 메모 수준으로 주장, 이유, 예시를 먼저 정리한다.
- 튜터 질문 예시:
  - 내 주장을 한 줄로 요약하면 무엇인가요?
  - 이유는 몇 가지로 나누는 게 가장 자연스러운가요?
  - PREP 구조가 맞나요, 발표형 구조가 맞나요?

### 26.3 Step 3. 영어 초안 작성
- 목표: 구조에 맞는 영어 초안을 만든다.
- 액션: 완벽한 문장보다 먼저 흐름이 맞는 초안을 만든다.
- 튜터 질문 예시:
  - 첫 문장이 주제를 분명히 보여주나요?
  - example 문장이 추상적이지 않고 구체적인가요?

### 26.4 Step 4. 문법·어휘·자연스러움 교정
- 목표: 문법 오류와 어색한 표현을 실제 영어답게 다듬는다.
- 액션: 시제, 주어-동사 일치, 연결어, 어휘 수준을 점검한다.
- 튜터 질문 예시:
  - 이 문장은 문법적으로 맞지만 원어민에게도 자연스러운가요?
  - 반복되는 단어를 더 적절한 표현으로 바꿀 수 있나요?

### 26.5 Step 5. 발표·면접·받아쓰기 대응 훈련
- 목표: 결과물을 실제 발화나 청취 상황에서도 사용할 수 있게 만든다.
- 액션: 발음 포인트, 끊어 읽기, 꼬리 질문, 받아쓰기 위험 구간을 정리한다.
- 튜터 질문 예시:
  - 이 문장은 말로 읽었을 때 너무 길지 않나요?
  - 선생님이 추가 질문을 하면 어떻게 이어서 답할 수 있나요?

## 28. 구현 우선순위 제안

실제로 `selfq`에 적용하려면 아래 순서가 가장 안전하다.

### Phase 1. Persona 정교화
- 국어, 수학, 영어 persona부터 확장
- processSteps, exampleTopics, starterTemplate, checkFocus 강화

### Phase 2. Result Model 분리
- 국어 `fullDraft`
- 수학 `solutionSteps`
- 영어 `speakingVersion`

처럼 과목별 결과물 구조를 분리

### Phase 3. Workflow JSON 표준화
- 모든 수행평가를 `assignment schema + workflow + result model` 구조로 통일

### Phase 4. Document Parser 연결
- 문서 분석 결과를 persona보다 먼저 반영
- 과목명만 보고 흐르는 구조를 줄이고
- 문서 형식 기반 분기 강화

## 29. 최종 정리

국어, 수학, 영어를 모두 풍성하게 만든다는 것은
과목 이름만 구분하는 것이 아니라,
각 과목이 문제를 풀고 글을 만들고 말을 준비하는 **사고 절차**를 엔진 안에 넣는 것이다.

따라서 다음 단계의 핵심은 이 세 가지다.

1. 과목별 persona 고도화
2. 과목별 workflow 표준화
3. 과목별 result model 분리

이 세 가지가 갖춰지면,
수행평가 자비스 엔진은 단순 생성기가 아니라
진짜 과목별 사고 파트너로 진화할 수 있다.
