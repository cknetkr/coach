# 자비스형 수행평가 AI 튜터 기획안

## 1. 프로젝트 개요

**프로젝트명**  
Jarvis for Schoolwork  
문서 해석형 수행평가 AI 튜터

**한 줄 정의**  
학생이 수행평가 문서나 과제 안내문을 넣으면, AI가 그 문서를 스스로 분석해 필요한 절차를 만들고, 단계별로 코칭하며, 최종 제출물까지 완성하도록 돕는 자율형 학습 에이전트

**핵심 목표**  
기존처럼 사람이 수행평가마다 절차를 미리 설계하는 것이 아니라,  
AI가 문서를 읽고 “무엇을, 어떤 순서로, 어떤 형식으로, 어떤 기준에 맞춰” 진행해야 하는지 스스로 만들어내는 시스템을 구축한다.

## 2. 문제 정의

현재 대부분의 AI 도구는 다음 한계를 가진다.

- 학생이 과제 형식을 직접 해석해야 한다.
- 수행평가마다 다른 절차를 사람이 다시 짜야 한다.
- 문서 형식이 바뀌면 프롬프트와 UI를 매번 수정해야 한다.
- AI는 결과 문장만 만들어 주고, 평가 기준 충족 여부를 구조적으로 관리하지 못한다.

즉, 지금 필요한 것은 “글을 써주는 AI”가 아니라  
**문서를 읽고, 절차를 설계하고, 수행을 안내하는 AI**다.

## 3. 핵심 철학

### 3.1 고정형이 아니라 해석형
이 서비스는 `수행1`, `수행2`, `수행3`을 미리 박아 두는 구조가 아니라,  
문서를 해석해서 새로운 과제 절차를 동적으로 생성해야 한다.

### 3.2 생성보다 진행이 중요
중요한 것은 최종 답안을 한 번에 출력하는 것이 아니라,
- 현재 단계가 무엇인지 파악하고
- 다음 질문을 만들고
- 부족한 항목을 보완하고
- 루브릭 충족 여부를 점검하는 것

즉, **AI는 작성기보다 튜터여야 한다.**

### 3.3 문서 중심 설계
모든 흐름의 시작점은 사용자의 자유 입력이 아니라 `과제 문서`다.  
문서를 기준으로 평가 항목, 제약 조건, 절차, 산출물 형식이 생성된다.

## 4. 서비스 목표

### 4.1 1차 목표
- 수행평가 문서를 읽고 항목과 조건을 자동 추출
- 항목을 기반으로 단계별 절차를 자동 생성
- 학생 답변을 받아 단계별 코칭 제공
- 최종 제출 형식으로 결과 정리

### 4.2 2차 목표
- 평가표, 루브릭, 예시 답안이 함께 있을 경우 더 정밀한 절차 생성
- 과목별로 다른 과제 유형에 대응
- 시험형 수행평가에서는 암기/내재화 훈련까지 연결

### 4.3 최종 목표
학생이 “이 과제 해야 해”라고 문서만 주면,
AI가 스스로
1. 과제를 해석하고
2. 절차를 만들고
3. 필요한 정보를 묻고
4. 중간 산출물을 정리하고
5. 제출 직전 점검까지 끝내는
**자율형 학습 에이전트**가 되는 것

## 5. 핵심 사용자 시나리오

### 시나리오 A: 문학 수행평가
- 학생이 수행평가 안내문 업로드
- AI가 `작품 선정 이유`, `줄거리`, `인물·배경`, `사회 연결`, `인용문` 등의 항목 추출
- 절차를 생성
- 학생에게 단계별 질문
- 최종 독서주제탐구서 작성

### 시나리오 B: 비평문 계획형
- AI가 `사회 문제 선정`, `원인 분석`, `자료 2건`, `해결 방안`, `구조 계획`을 추출
- 단계별 계획 수립
- 참고자료와 검증 과정까지 코칭
- 최종 비평문 개요 완성

### 시나리오 C: 시험장 작성형
- AI가 `1500자`, `5문단`, `자료 반입 불가` 조건을 해석
- 원고 작성 지원
- 이후 암기 카드, 빈칸 훈련, 재현 테스트 제공

## 6. 핵심 차별점

### 기존 방식
- 사람이 절차를 미리 정의
- AI는 정해진 입력창만 채워줌

### 제안 방식
- AI가 문서를 해석
- 절차를 자동 생성
- 절차 중 부족한 부분을 재조정
- 과제 형식이 바뀌어도 엔진이 대응

즉, 이 서비스의 차별점은 `답변 생성 능력`보다  
**절차 생성 능력**에 있다.

## 7. 시스템 구조

### 7.1 전체 구조
`문서 입력 → 문서 해석 → 절차 생성 → 단계별 튜터링 → 결과 검증 → 제출본/훈련본 생성`

### 7.2 핵심 엔진

#### A. Document Parser
역할:
- 문서에서 과제명, 항목명, 순서, 필수 조건, 제약, 산출물 형식을 추출

추출 대상:
- 항목 제목
- 세부 설명
- 필수 여부
- 최소 개수
- 금지 조건
- 글자 수
- 참고자료 요구
- 평가 기준

예시:
- “인상적인 문구 2개, page 번호 필수, 내용 겹치지 않게”
- “1500±50자”
- “도입-전개1-전개2-전개3-정리”

#### B. Workflow Generator
역할:
- 추출된 정보를 실제 학습 절차로 변환

예시:
- Step 1: 작품 선정 이유 작성
- Step 2: 인물과 배경 분석
- Step 3: 줄거리 정리
- Step 4: 사회 문제 연결
- Step 5: 인용문 2개 선택 및 비교 검토

특징:
- 과제마다 다른 단계 수 허용
- 단계 내 세부 필드 자동 생성
- 누락 조건이 있으면 보완 스텝 자동 추가

#### C. Tutor Orchestrator
역할:
- 각 단계에서 학생과 대화
- 다음 질문 생성
- 학생 답변 수준에 따라 피드백 제공
- 바로 다음 행동 제안

예시:
- “사회 현실 연결이 약합니다. 기사나 통계 근거를 넣어볼까요?”
- “인물 분석은 있는데 시대 배경이 빠졌어요.”

#### D. Rubric Checker
역할:
- 현재 결과물이 문서 조건을 얼마나 충족하는지 판단
- 누락 항목, 약한 항목, 감점 위험 요소 표시

검사 예:
- 개수 충족 여부
- 글자 수 범위
- 인용문 중복 여부
- 문단 구조 충족 여부
- 참고자료 수 충족 여부

#### E. Output Composer
역할:
- 단계별 입력을 최종 제출 형식으로 재조합

출력 형태:
- 독서기록서
- 비평문 계획서
- 완성 비평문
- 발표 대본
- 암기 카드

#### F. Internalization Trainer
역할:
- 시험형 과제를 위해 완성본을 암기/재현 가능한 형태로 변환

기능:
- 문단 요약
- 핵심 키워드 카드
- 빈칸 문제
- 재현도 비교
- 시간 시뮬레이션

## 8. 데이터 모델 설계

AI가 문서를 해석한 뒤 내부적으로 아래 같은 구조를 만들어야 한다.

```json
{
  "assignmentName": "독서주제탐구",
  "assignmentType": "reading-analysis",
  "steps": [
    {
      "id": "reason",
      "title": "작품 선정 이유",
      "required": true,
      "inputType": "textarea",
      "constraints": ["구체성", "사회적 연결"],
      "coachGoals": ["왜 이 작품을 골랐는지 분명해야 함"]
    },
    {
      "id": "quotes",
      "title": "인상적인 문구",
      "required": true,
      "repeat": 2,
      "subfields": ["page", "quote", "reason"],
      "constraints": ["서로 다른 내용", "page 번호 필수"]
    }
  ],
  "globalConstraints": [
    "책 전체 흐름이 드러나야 함"
  ],
  "finalOutput": {
    "type": "worksheet"
  }
}
```

중요한 점은 이 JSON을 사람이 직접 쓰는 게 아니라,  
**AI가 문서로부터 자동 생성**해야 한다는 것이다.

## 9. 튜터링 흐름 설계

### 9.1 기본 대화 흐름
1. 문서 확인
2. 과제 구조 추출
3. 절차 생성
4. 사용자에게 현재 단계 제시
5. 단계별 입력 수집
6. 중간 검토
7. 최종 정리
8. 제출 전 체크

### 9.2 튜터 질문 원칙
- 한 번에 한 단계만 묻는다.
- 질문은 문서 요구사항과 직접 연결돼야 한다.
- 모호한 답이 오면 자동으로 구체화 질문을 던진다.
- 다음 단계로 넘어가기 전 최소 조건을 확인한다.

### 9.3 피드백 원칙
- “좋아요”보다 “무엇이 부족한지”를 구체적으로 말한다.
- 감점 위험을 먼저 보여준다.
- 바로 수정 가능한 문장 단위 피드백을 준다.

대응:
- 최초 생성 후 “AI가 해석한 과제 구조”를 사용자에게 요약 확인
- 사용자가 수정 가능하게 함

### 리스크 4: 글 생성기로 오해될 수 있음
대응:
- 시스템 목적을 “대필”이 아니라 “코칭”으로 분명히 설계
- 사고 과정, 근거, 검증 흐름을 강조

## 14. 핵심 결론

이 프로젝트의 본질은 수행평가 앱이 아니다.  
**문서를 읽고 스스로 절차를 설계하는 자비스형 학습 에이전트**다.

즉, 성공 여부는 예쁜 화면보다 다음 능력에 달려 있다.

- 문서를 읽는 능력
- 요구사항을 구조화하는 능력
- 절차를 생성하는 능력
- 단계별로 대화를 이어가는 능력
- 마지막에 평가 기준으로 검증하는 능력

## 16. 기존 SelfQ 수행평가 방식 분석

현재 `selfq`의 수행평가 생성 구조는 완전히 단순한 초안 생성기가 아니라, 이미 다음과 같은 흐름을 갖고 있다.

1. 수행평가 전용 온보딩 진입
- 일반 시험 생성 플로우와 분리된 `perf_collect` 경로로 들어간다.
- 이 단계에서 수행평가 유형, 주제, 루브릭, 첨부파일을 수집한다.

2. 수행평가용 공부방 생성
- 수집된 정보를 바탕으로 수행평가 전용 방을 만든다.
- 방에는 `perf_type`, `perf_topic`, `perf_rubric`, `constraints`, `persona` 같은 정보가 저장된다.

3. 과목별 페르소나 기반 절차 생성
- Gemini 프롬프트에서 과목별 `processSteps`, `discoveryPrompt`, `exampleTopics`, `starterTemplate`를 사용한다.
- 이를 바탕으로 `summary`, `constraints`, `initialQuestions`, `steps[]` 구조의 plan을 생성한다.

4. 단계형 과외 진행
- 생성된 step마다 사용자에게 필요한 질문을 먼저 던진다.
- 각 단계가 끝날 때 step 결과물, 팁, self-check, 다음 단계 힌트를 생성한다.

5. 최종 결과물 보완
- 전체 초안, 루브릭, 체크리스트, 코칭 메시지를 다시 생성하거나 보완한다.
- 필요하면 AI로 초안을 수정 적용하고 결과를 저장한다.

즉, 현재 `selfq`는 이미  
**“루브릭/문서 입력 → 단계 생성 → 단계별 코칭 → 초안 보완”**  
구조를 어느 정도 구현하고 있다.

## 17. 기존 방식과 새 기획안 비교 분석

### 17.1 기존 SelfQ 방식의 강점
- 수행평가를 일반 시험 생성 흐름과 분리해 전용 경로로 설계했다.
- 과목별 페르소나를 두어 국어, 영어, 수학, 과학 등에서 다른 절차를 적용한다.
- `steps`, `questions`, `constraints`를 생성하는 구조가 이미 존재한다.
- 단계별 빌드업 방식이라 학생이 한 번에 완성하지 않아도 된다.
- 루브릭, 초안, 체크리스트, 코칭을 분리하여 품질을 높일 수 있다.

### 17.2 기존 SelfQ 방식의 한계
- 절차 생성의 출발점이 `문서 자체`보다 `과목별 사전 정의 프로세스`에 더 가깝다.
- 새로운 수행평가 형식이 들어오면 페르소나의 `processSteps`에 크게 의존한다.
- 문서가 완전히 새로운 형식일 경우 절차를 스스로 새로 발명하기보다 기존 과목 템플릿에 맞춰 해석할 가능성이 높다.
- 온보딩에서는 아직 “절차형 과외”보다 “공부방 생성” 흐름이 먼저 보인다.
- 문서 해석 결과가 독립적인 표준 데이터 모델로 축적되기보다, 프롬프트 내부에서 바로 소비되는 비중이 크다.

### 17.3 새 기획안의 차별점
새 기획안은 기존 방식보다 한 단계 더 나아가,  
**과목 페르소나 기반 절차 생성**에서  
**문서 해석 기반 절차 생성**으로 중심축을 이동시키는 것을 목표로 한다.

기존 방식:
- 과목별 `processSteps`가 먼저 있음
- 문서와 루브릭은 그 절차를 구체화하는 재료로 사용됨

새 기획안:
- 문서가 먼저 들어옴
- 문서에서 항목, 순서, 개수, 금지 조건, 산출물 형식을 추출
- 그 결과로 절차를 생성
- 과목 페르소나는 표현 방식과 피드백 톤을 보강하는 보조 엔진으로 사용

즉, 기존 방식은  
`과목 중심 절차 생성`이고,  
새 기획안은  
`문서 중심 절차 생성`이다.

## 18. 고도화 방향 제안

### 18.1 유지해야 할 자산
기존 `selfq`에서 다음 자산은 유지 가치가 높다.

- 수행평가 전용 온보딩 분기 구조
- 과목별 페르소나 시스템
- 단계별 Q&A + build step 구조
- rubric / draft / checklist / customGuide 분리 구조
- `PerfGuidedFlow` 같은 대화형 빌드업 UX

이 부분은 새 기획안에서도 버릴 것이 아니라, 핵심 기반으로 삼아야 한다.

### 18.2 바꿔야 할 중심 엔진
고도화의 핵심은 `callGeminiPerfAnalyze` 이전에  
별도의 `Document Parser` 레이어를 두는 것이다.

기존:
- subjectName + perfType + rubricText + 파일
- 바로 Gemini 프롬프트로 전달
- 프롬프트 안에서 steps/questions 생성

개선:
- 문서 파싱
- 항목/조건/제약 추출
- 구조화된 assignment JSON 생성
- assignment JSON + 과목 페르소나를 함께 Gemini에 전달
- 이후 step/questions 생성

이렇게 되면 절차 생성의 근거가 더 명확해지고,
새로운 수행평가 형식에도 더 잘 적응할 수 있다.

### 18.3 추천 아키텍처

1. `Document Parser`
- 수행평가 문서에서 항목, 제약, 루브릭, 산출물 형식 추출

2. `Assignment Schema Builder`
- 추출 결과를 공통 JSON 구조로 변환

3. `Workflow Generator`
- JSON을 바탕으로 단계, 질문, 체크포인트 생성

4. `Persona Layer`
- 과목 특성에 맞는 표현, 예시 주제, 피드백 스타일 보강

5. `Guided Tutor`
- 단계별 Q&A와 빌드업 진행

6. `Result Composer`
- 최종 제출본, 체크리스트, 내재화 훈련 결과 생성

## 19. 결론

기존 `selfq`의 수행평가 구조는 이미 훌륭한 출발점이다.  
특히 과목별 페르소나와 단계형 빌드업 구조는 자비스형 학습 시스템의 핵심 자산이다.

하지만 현재 구조는 아직  
**“문서를 읽고 스스로 절차를 발명하는 시스템”** 이라기보다  
**“과목별 기본 절차를 가지고 문서에 맞춰 조정하는 시스템”** 에 가깝다.

새 기획안은 이 차이를 메우기 위한 방향이다.  
즉, `selfq`의 장점은 계승하되,
절차 생성의 중심을 과목 템플릿에서 문서 해석 엔진으로 이동시키는 것이 핵심이다.

한 줄로 정리하면:

**기존 SelfQ는 과목 중심 수행평가 코치이고,  
새 기획안은 문서 중심 자율형 수행평가 자비스다.**

## 20. 최근 국어 업데이트 분석

최근 `selfq` 국어 업데이트는 단순히 프롬프트 문장을 바꾼 수준이 아니라,  
국어 수행평가 전용 페르소나와 흐름을 실제 생성 구조에 더 깊게 연결한 변경으로 볼 수 있다.

### 20.1 이번 업데이트의 핵심 변화

1. 국어 전용 페르소나 강화
- `국어`, `문학`, `독서`, `글쓰기`, `비평`, `논술` 키워드에 반응하는 전용 페르소나가 추가·강화되었다.
- 페르소나 명칭은 `수석 논술 코치 (문학·국어 수행평가 전담)`으로 설정되어 있다.
- 목표는 단순 감상문이 아니라 `문단 구조`, `분량`, `사회 문제 연계`, `루브릭 충족` 중심의 고득점 비평문 완성이다.

2. 국어 수행평가용 절차 명시
- processSteps가 다음처럼 구조화되었다.
  - 도서 분석
  - 사회 문제 매핑 및 개요 작성
  - 초안 작성
  - 조건 검증 및 글자 수 조정
- 즉, 국어 과제는 이제 “읽고 느낀 점 쓰기”가 아니라
  **작품 분석 → 현실 연결 → 논증 구조화 → 조건 검증**
  흐름으로 명확히 정의되었다.

3. 예시 주제와 시작 템플릿 보강
- `학교 폭력과 방관자 문제`, `환경 위기와 개인 책임`, `청소년 소외와 관계 단절` 같은 예시 주제가 추가되었다.
- `현재 단계: [도서 분석] -> [사회 문제 매핑] -> [개요 작성] -> [초안 작성] -> [조건 검증] -> [글자 수 조정]`
  같은 starter template이 들어가면서, AI가 국어 과제를 시작할 때 기본 사고 흐름을 바로 갖게 되었다.

4. 단계형 결과 저장 개선
- `PerfGuidedFlow`의 최종 조립 단계에서 `draftSections`, `fullDraft`, `stepResults`, `finalComment`까지 저장하도록 확장되었다.
- 이 변경으로 국어 수행평가에서 중요한 `문단별 구조`, `통합 초안`, `단계별 축적 결과`를 한꺼번에 관리하기 쉬워졌다.

5. 방 내부 보강 루프 강화
- `SubjectStudyView`에서는 이제 수행평가 방 안에서
  - 페르소나 테스트
  - 실전 생성 미리보기
  - AI 코칭
  - AI 수정 적용
  까지 이어진다.
- 국어처럼 초안 수정이 중요한 과목에서 특히 유리한 구조다.

## 21. 국어 업데이트 기준 현재 방식의 흐름 분석

이번 국어 업데이트 이후 현재 `selfq`의 국어 수행평가 흐름은 아래처럼 정리할 수 있다.

### 21.1 현재 실행 흐름

1. 수행평가 생성 시작
- 학생이 수행평가 카테고리로 진입한다.
- 유형, 주제, 루브릭, 첨부파일을 수집한다.

2. 공부방 생성
- 국어 수행평가 전용 방이 생성된다.
- 이때 주제, 루브릭, 첨부 수, persona 정보가 함께 저장된다.

3. 국어 페르소나 기반 계획 생성
- AI는 국어 페르소나의 processSteps를 참고해 프로젝트 plan을 만든다.
- 이때 summary, constraints, initialQuestions, steps가 생성된다.

4. 단계별 과외
- 1단계: 작품/도서 분석
- 2단계: 사회 문제 연결과 개요 설계
- 3단계: 초안 작성
- 4단계: 조건 검증 및 분량 조정
- 각 단계 전후로 필요한 질문과 결과물이 생성된다.

5. 최종 초안 및 코칭
- 생성된 결과를 통합 초안으로 모은다.
- AI 코칭이나 수정 적용을 통해 제출 수준까지 다듬는다.

### 21.2 이번 업데이트가 의미하는 것

이 구조는 국어 수행평가를 단순 생성형 과제가 아니라  
**“작품 해석과 논증형 글쓰기 프로젝트”** 로 본다는 점에서 의미가 크다.

특히 다음 두 가지가 중요하다.

- 국어 수행평가의 본질을 `사회 문제 연결형 비평문` 중심으로 해석하고 있다.
- 초안 생성보다 `단계별 사고 흐름`과 `조건 검증`을 강조한다.

이는 기존보다 훨씬 자비스형 구조에 가까운 발전이다.

## 22. 국어 업데이트 기준 기존 방식과 새 기획안 재비교

### 22.1 기존 SelfQ 국어 방식
- 과목이 국어로 분류되면 국어 전용 페르소나를 선택한다.
- 그 페르소나에 미리 정의된 단계에 따라 절차를 구성한다.
- 문서는 루브릭과 제약 조건을 보완하는 자료로 사용된다.
- 결과적으로 국어 비평문에 최적화된 고정형-반동적 구조다.

### 22.2 새 기획안 관점에서 본 한계
- 국어 수행평가의 절차가 여전히 `국어라는 과목명`에 크게 묶여 있다.
- 예를 들어 같은 국어라도
  - 독서 기록장
  - 독후감
  - 발표 대본
  - 토론 개요서
  - 서평
  - 비평문
  은 절차가 달라질 수 있는데,
  현재 구조는 이 차이를 문서보다 과목 중심으로 처리할 가능성이 높다.

즉,
- 현재 방식은 `국어 = 비평문 중심 절차`
- 새 기획안은 `문서가 요구하는 국어 과제 형식에 따라 절차를 다시 생성`

이라는 차이가 있다.

### 22.3 국어 업데이트가 새 기획안에 주는 시사점

이번 국어 업데이트는 새 기획안에 매우 중요한 힌트를 준다.

1. 과목별 페르소나는 필요하다
- 완전 범용 문서 해석만으로는 과목 특유의 좋은 질문과 좋은 흐름을 만들기 어렵다.
- 국어처럼 비평문·독서·논술의 결이 중요한 과목은 전문 페르소나가 큰 도움이 된다.

2. 하지만 페르소나만으로는 부족하다
- 현재 국어 프로세스는 강력하지만, 여전히 비평문 중심 편향이 있다.
- 앞으로는 국어 내부에서도 과제 유형별 절차 분기가 더 필요하다.

3. 최적 구조는 하이브리드다
- 1차: 문서에서 과제 형식 추출
- 2차: 국어 페르소나가 그 형식에 맞는 표현과 코칭 방식 보강
- 3차: 최종 workflow 생성

즉, 새 기획안은 기존 `selfq`의 국어 업데이트를 부정하는 것이 아니라,
그 구조를 더 일반화하고 더 정교하게 확장하는 방향이어야 한다.

## 23. 보완된 기획 방향

국어 업데이트를 반영한 뒤의 최적 기획 방향은 다음과 같다.

### 23.1 1단계: 문서에서 국어 과제 형식 추출
- 비평문인지
- 독서주제탐구인지
- 발표문인지
- 서평인지
- 감상문인지
- 토론 준비서인지

를 먼저 판단한다.

### 23.2 2단계: 국어 페르소나 적용
- 작품 분석 방식
- 사회 문제 연결 방식
- 개요 설계 방식
- 문단 검증 방식

을 과목 전문성에 맞게 보강한다.

### 23.3 3단계: 형식 맞춤 workflow 생성
예:
- 독서주제탐구: 작품 선정 이유 → 인물·배경 → 줄거리 → 태도 변화 → 사회 연결
- 비평문: 작품 해석 → 사회 문제 매핑 → 개요 → 초안 → 분량 검증
- 발표문: 핵심 주장 → 말하기 구조 → 발표용 문장 → 시간 검증

### 23.4 4단계: 결과물 + 훈련 분리
- 제출용 결과물 생성
- 발표/시험 대비용 내재화 훈련 생성

## 24. 최종 해석

최근 국어 업데이트는 `selfq`가 이미
**과목별 페르소나 기반 절차 생성기**로 진화하고 있음을 보여 준다.

특히 국어는
- 작품 분석
- 사회 문제 연결
- 문단 구조
- 조건 검증

이라는 흐름이 명확해져서,
자비스형 수행평가 AI의 가장 앞선 사례로 볼 수 있다.

하지만 다음 단계는 여기서 멈추지 않는다.
앞으로는
**국어라는 과목 단위 절차**를 넘어서
**국어 안의 과제 문서 형식별 절차 생성**
으로 발전해야 한다.

한 줄로 정리하면:

**이번 국어 업데이트는 좋은 과목 중심 튜터를 만들었고,  
새 기획안은 그것을 문서 중심 자비스로 확장하는 다음 단계다.**
